🗺️ Visão Geral do Fluxo

O objetivo deste documento é apresentar o fluxo de aprovação para as pull requests realizadas pela equipe de desenvolvimento do ateliê de software alocado na SME, definindo quem pode e irá realizar as aprovações de alterações ou criações de códigos no GitHub.

ℹ️
Fluxo de publicação:
Desenvolvimento (Local) → QA → Homologação → (Treinamento, se necessário) → Produção

💻 Ambiente de Desenvolvimento Local

  • Ambiente local, executado na máquina do desenvolvedor
  • Desenvolvedores criam feature branches a partir da main
  • Liberdade para testes iniciais e experimentação

Transição para QA

🔀
  • O desenvolvedor realiza um Pull Request (PR) para o ambiente de QA
  • PR deve ser aprovada por um Desenvolvedor Sênior e/ou Líder Técnico
  • Após aprovação, a branch é mergeada em teste e o deploy automático é realizado

🧪 Ambiente de Testes e Qualidade (QA)

Atividades no ambiente

  • Testes funcionais e manuais
  • Testes de regressão
  • Validação de regras de negócio
  • Análise de comportamento esperado
Responsáveis
QA (execução) QA Sênior (validação final) Desenvolvedor
🔷 Transição para Homologação
  • Após testes executados e aprovados, o desenvolvedor abre um PR para Homologação
  • O PR deve ser aprovado por um QA Sênior
  • Apenas PRs vindo da branch teste são aceitas em homologação
  • Revisão obrigatória do Quality Gate antes do deploy
  • Deploy automático após merge
  • Se reprovado → retorna para desenvolvimento

🔷 Ambiente de Homologação

Características do ambiente

  • Ambiente estável
  • Similar ao ambiente produtivo
  • Utilizado para validação funcional e de negócio
  • Ambiente oficial de apresentação na Review para validação do PO
Responsáveis
QA Desenvolvedor Sênior PO (Product Owner)
⚠️
Decisão Importante
Neste ponto define-se se o desenvolvimento precisa ser apresentado a usuários finais, além do PO na review.
🚀 Aceite do Produto (PO)

Caso não seja necessário o ambiente de Treinamento, o código deve receber o aceite formal do PO ainda em Homologação para seguir para Produção.

🎓 Ambiente de Treinamento Opcional

  • Treinamentos formais
  • Apresentações institucionais
  • Validações com grupos controlados de usuários

Publicação em Treinamento

📋
  • O código deve ser publicado a partir da Homologação
  • A publicação deve ser realizada exclusivamente por um Desenvolvedor Sênior ou Líder Técnico

🚀 Ambiente de Produção

Pré-requisitos obrigatórios

Testes
✅ Concluídos e aprovados
Aceite do PO
✅ Registrado formalmente
Código
✅ Validado em Homologação

Regras de publicação

  • Apenas por Desenvolvedor Sênior ou Líder Técnico
  • Apenas PRs vindo da branch homologação são aceitas
  • Revisão obrigatória do Product Owner antes do merge em prod
  • Quality Gate reprovado → devolve para feature e segue todo o caminho novamente
  • Deploy automático após merge
🚫
Quality Gate reprovado: O desenvolvimento retorna para feature e deve seguir todo o caminho novamente: feature → teste → homologação → produção.
